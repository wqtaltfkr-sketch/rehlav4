# رحلة الحياة الزوجية — PWA

منصة تعليمية تفاعلية بأربع مراحل (ما قبل الخطوبة، الخطوبة، حديثو الزواج، الاستقرار الأسري)، مبنية بـ HTML/CSS/JS خالص (بدون أي إطار عمل) + Supabase، وجاهزة للعمل كتطبيق PWA على الكمبيوتر والجوال والتابلت.

## 📁 محتويات المشروع

## 🚀 خطوات التشغيل (3 خطوات فقط)

### 1) قاعدة البيانات (Supabase)
1. أنشئ مشروعاً جديداً على [supabase.com](https://supabase.com)
2. اذهب إلى **SQL Editor** ونفّذ ملفات `schema*.sql` بالترتيب التالي بالضبط (كل ملف يعتمد على الذي قبله):
   1. `schema.sql` (الأساسي: profiles، contents، user_progress، journal_entries، تذاكر الدعم)
   2. `schema_admin.sql` (جداول فقرات/وسائط الدروس وصلاحيات الإدارة الإضافية)
   3. `schema_ai.sql` (سجل أسئلة مساعد الطالب الذكي)
   4. `schema_ai_v2.sql` (تقييم إجابات المساعد الذكي 👍/👎)
   5. `schema_notifications.sql` (جداول نظام الإشعارات Push)
   6. `schema_media_transcript.sql` (دعم فهم نص الفيديو للمساعد الذكي)
   7. `schema_fixes_v2.sql` (تفعيل Realtime تلقائياً + دوال تشخيص الإدارة)
   8. `schema_roles.sql` 🆕 (نظام الأدوار user/supervisor/advisor — Sprint 1)
   9. `schema_checklists.sql` 🆕 (قوائم تجهيزات المرحلة — Sprint 2)
3. من **Settings → API** انسخ: `Project URL` و `anon public key`

### 2) الإعدادات
افتح `js/config.js` وعدّل:
```js
SUPABASE_URL: "https://xxxx.supabase.co",
SUPABASE_ANON_KEY: "eyJhbGciOi...",
WHATSAPP_DEFAULT_NUMBER: "201000000000", // بصيغة دولية بدون + أو صفر
```

### 3) الرفع والنشر (GitHub + Cloudflare Pages)
```bash
git init
git add .
git commit -m "أول نسخة من منصة رحلة الحياة الزوجية"
git branch -M main
git remote add origin https://github.com/USERNAME/REPO.git
git push -u origin main
```
ثم في [Cloudflare Pages](https://dash.cloudflare.com) (هذا هو الاستضافة الفعلية المستخدمة لهذا المشروع):
1. **Workers & Pages → Create → Pages → Connect to Git**
2. اختر المستودع من GitHub
3. **Build settings**: لا حاجة لأي أمر Build (الموقع Static بالكامل) — اترك **Build command** فارغاً، و**Build output directory** = `/` (أو اتركه فارغاً)
4. اضغط **Save and Deploy**
5. تأكد من إضافة دومين Cloudflare الفعلي (مثال: `yourapp.pages.dev`، وأي دومين مخصص لاحقاً) إلى **Supabase Dashboard → Authentication → URL Configuration** في كل من **Site URL** و **Redirect URLs**، وإلا فشلت روابط تأكيد البريد واستعادة كلمة المرور بصمت.

بعد أول نشر، أي `git push` جديد على الفرع الرئيسي سيُحدّث الموقع تلقائياً (Auto-deploy مفعّل افتراضياً في Cloudflare Pages).

### 🆕 فحص تلقائي قبل كل نشر (CI)
أضفنا `.github/workflows/ci.yml` يعمل تلقائياً عند كل `push` أو Pull Request، ويقوم بخطوتين:
1. **فحص صياغة كل ملفات JavaScript** (`node --check`) — يوقف أي خطأ صياغي قبل وصوله للإنتاج.
2. **تنفيذ كل ملفات `schema*.sql` بالترتيب الرسمي** على قاعدة بيانات PostgreSQL مؤقتة داخل GitHub Actions نفسه — يكتشف فوراً أي خطأ SQL أو تبعية ناقصة بين الملفات قبل أن تحتاج لصقها يدوياً في Supabase. لا يلمس هذا الفحص مشروع Supabase الحقيقي إطلاقاً (قاعدة بيانات مؤقتة تُحذف فور انتهاء الفحص).

يمكن متابعة نتيجة الفحص من تبويب **Actions** في مستودع GitHub بعد كل رفعة.

## 🛠️ إصلاحات مهمة على من لديه مشروع منشور مسبقاً (Upgrade)
إن كنت قد نفّذت `schema.sql` أو رفعت الواجهة **قبل** هذا التحديث، نفّذ الخطوتين التاليتين مرة واحدة فقط:

1. **تفعيل الرد الفوري (Realtime) في الدعم الفني:** نفّذ في SQL Editor:
   ```sql
   ALTER PUBLICATION supabase_realtime ADD TABLE ticket_messages;
   ```
   (بدون هذا السطر، محادثة الدعم تعمل لكن الرسائل الجديدة لا تظهر فوراً وتحتاج تحديث الصفحة يدوياً — لا تظهر أي رسالة خطأ، الميزة تفشل بصمت).
2. **استعادة كلمة المرور:** ارفع أحدث نسخة من `index.html` و `js/app.js` و `js/auth.js`. قبل هذا التحديث كان زر "نسيت كلمة المرور؟" يُرسل بريداً فعلاً، لكن لا توجد شاشة لإدخال كلمة مرور جديدة بعد فتح الرابط. الآن تظهر شاشة "تعيين كلمة مرور جديدة" تلقائياً عند فتح الرابط من البريد (تُكتشف عبر حدث `PASSWORD_RECOVERY` الرسمي من Supabase وليس بتحليل الرابط يدوياً).
   - ⚠️ تذكّر رفع رقم `CACHE_NAME` في `service-worker.js` بعد هذا التحديث (v5 → v6) حتى لا يستمر متصفح المستخدمين العائدين في تقديم نسخة قديمة مخزّنة من الملفات.

## 👤 تعيين مستخدم كمشرف أو مستشار (Roles) — 🆕 مُحدَّث في Sprint 1
بعد تنفيذ `schema_roles.sql`، أصبح لدى كل مستخدم عمود `role` (`user` / `supervisor` / `advisor`) بجانب عمود `is_supervisor` القديم (لا يزال يعمل تماماً — الاثنان متوافقان معاً).

**الطريقة الموصى بها الآن (من الواجهة مباشرة):** من تبويب **👥 المستخدمون** في لوحة تحكم الإدارة، اختر الدور الجديد من القائمة المنسدلة بجانب اسم المستخدم واضغط **حفظ الدور** — لا حاجة للدخول إلى Supabase يدوياً بعد الآن (يتطلب أن يكون حسابك أنت بالفعل مشرفاً).

**الطريقة اليدوية (أول مشرف على الإطلاق، أو كبديل):** من **Table Editor → profiles** غيّر عمود `role` إلى `supervisor` (أو `is_supervisor` إلى `true` كما في السابق تماماً) للمستخدم المطلوب. بمجرد تسجيل دخوله من جديد، سيظهر له تبويب "🛠️ الإدارة" تلقائياً في القائمة.

الدور `advisor` (مستشار) محجوز حالياً لتوسعات مستقبلية (صلاحيات أدق من "مشرف" كامل)، ولا يمنح حالياً أي صلاحية إضافية عن مستخدم عادي في الواجهة.

## 🛠️ لوحة تحكم الإدارة (Admin Dashboard)
متاحة فقط لمن `is_supervisor = true`، عبر مسار `#admin` (محمي في الواجهة وبـ RLS في قاعدة البيانات معاً). تشمل 6 تبويبات:

1. **📊 الإحصائيات** — إجمالي المستخدمين (ذكور/إناث)، إجمالي الدروس، الدروس المكتملة، تذاكر الدعم المفتوحة.
2. **📚 المحتوى** — إنشاء/تعديل/حذف الدروس بالكامل، ولكل درس: عناوين فرعية وفقرات نصية غير محدودة (`content_sections`) وروابط فيديو يوتيوب/PDF/روابط عامة (`content_media`). لا حاجة للدخول إلى Supabase يدوياً بعد الآن.
3. **👥 المستخدمون** — عرض كل المستخدمين وتعديل مرحلتهم الحالية مباشرة.
4. **💬 الدعم** — عرض كل تذاكر الدعم، فتح تذكرة جديدة نيابةً عن أي مستخدم، الرد الفوري (Realtime) داخل التذكرة، إغلاق/إعادة فتح التذكرة، أو حذفها نهائياً.
5. **🔔 الإشعارات** — إرسال تذكير يدوي (لكل المستخدمين أو لمستخدم معيّن)، تشغيل تذكير فوري لمن لم يُكمل درساً منذ عدد أيام محدد، وجدول بآخر 20 إشعاراً مُرسلاً. التفاصيل الكاملة في القسم التالي.
6. **📤 تصدير البيانات** — تصدير أي جدول (مستخدمين، دروس، فقرات، وسائط، تقدّم، تذاكر، رسائل) كملف CSV، أو تنزيل نسخة احتياطية كاملة بصيغة JSON لكل الجداول دفعة واحدة.

**ملاحظة خصوصية مهمة:** جدول `journal_entries` (المذكرات الخاصة) مستثنى عمداً من كل صلاحيات المشرف وأدوات التصدير — لا يمكن لأي مشرف قراءتها أو تصديرها، بالضبط كما في تصميم المشروع الأصلي.

### تدفق إضافة/تعديل درس جديد الآن
1. تبويب **المحتوى** → **+ درس جديد** → إدخال العنوان/المرحلة/التصنيف/الجنس ثم **حفظ بيانات الدرس**.
2. بعد الحفظ يظهر قسم "الفقرات والعناوين" و"روابط الفيديو والملفات" — أضف كل ما تريد وستظهر تلقائياً بترتيبها عند فتح المستخدم للدرس.

## ✅ قائمة تجهيزات المرحلة (Checklist) — 🆕 Sprint 2
شاشة جديدة "قائمة التجهيزات" (`#checklist`) تغطي فجوة كانت موثّقة في التقرير التحليلي: لا يوجد حالياً أي أداة عملية غير الدروس النصية لمرحلة "الخطوبة" تحديداً.

**كيف تعمل:**
- بعد تنفيذ `schema_checklists.sql`، تُزرَع تلقائياً 6 بنود افتراضية لمرحلة "الخطوبة" (ميزانية، حجز قاعة، قائمة مدعوين، أوراق رسمية، مسكن الزوجية، فحص طبي) — يمكن للمشرف لاحقاً إضافة/حذف بنود افتراضية أخرى مباشرة من **Table Editor → checklist_items** (`is_custom = false`).
- كل مستخدم يرى بنود مرحلته الحالية فقط، ويؤشّر عليها ✓ عند إنجازها (وله حرية إلغاء التأشير في أي وقت، بخلاف الدروس).
- يمكن لأي مستخدم إضافة بنوده الخاصة به فوق البنود الافتراضية (مثال: "التنسيق مع المصور")، وحذفها لاحقاً — هذه البنود الخاصة لا تظهر لأي مستخدم آخر.
- شريط تقدّم بسيط أعلى الشاشة يعرض نسبة إنجاز القائمة الحالية.

**الخصوصية:** بخلاف المذكرات الشخصية، بنود القائمة (حتى الخاصة منها) يمكن للمشرف رؤيتها لأغراض الدعم الفني (تفصيل ذلك موثّق داخل `schema_checklists.sql` نفسه، مع تعليمات لتغيير هذا السلوك إن رغبتم).

## 🤖 مساعد الطالب الذكي (سؤالك عن الدرس) — Mistral AI

زر عائم يظهر فقط داخل صفحة "الدرس النشط"، يفتح نافذة سؤال وجواب: الطالب يسأل عن أي جزء من الدرس، والمساعد يرد **بالاعتماد فقط على محتوى هذا الدرس** (فقراته وعناوينه ووسائطه المسجّلة فعلياً في قاعدة البيانات) مع ذكر مصدر كل إجابة (مثال: "الفقرة رقم 2"، أو اسم الفيديو). لو لم تكن الإجابة موجودة في الدرس، يصرّح المساعد بذلك ويقترح التواصل مع المشرف عبر صفحة الدعم الفني، بدل اختلاق إجابة.

### كيف يعمل تقنياً
```
الطالب (المتصفح) → Supabase Edge Function (ask-lesson-ai) → Mistral API
                                    ↓
                     يبني سياق نصي من contents + content_sections + content_media
                     الخاصة بالدرس المفتوح فقط، ثم يرسله للنموذج مع تعليمات صارمة
                     بعدم الإجابة من خارج هذا السياق.
```
مفتاح Mistral API **لا يوضع أبداً في كود الواجهة** — يبقى سرّاً على خادم Supabase (Edge Function) فقط، وهذا هو السبب في استخدام Edge Function بدل استدعاء Mistral مباشرة من المتصفح.

### خطوات التفعيل

**1. قاعدة البيانات** — بعد `schema.sql` و`schema_admin.sql`، نفّذ أيضاً `schema_ai.sql` في SQL Editor (ينشئ جدول `lesson_ai_questions` لتسجيل الأسئلة وتطبيق حدّ الاستخدام).

**2. احصل على مفتاح Mistral API**
من [console.mistral.ai](https://console.mistral.ai) → API Keys → أنشئ مفتاحاً جديداً.

**3. ثبّت Supabase CLI واربطه بمشروعك** (مرة واحدة فقط)
```bash
npm install -g supabase
supabase login
supabase link --project-ref YOUR-PROJECT-REF   # موجود في رابط مشروعك على Supabase
```

**4. خزّن مفتاح Mistral كسرّ (Secret) على خادم Supabase**
```bash
supabase secrets set MISTRAL_API_KEY=ضع_مفتاحك_هنا
```

**5. انشر الدالة (Edge Function)**
```bash
supabase functions deploy ask-lesson-ai
```
بعد النشر تعمل الدالة تلقائياً على:
`https://YOUR-PROJECT-REF.supabase.co/functions/v1/ask-lesson-ai`
ولا حاجة لتعديل أي شيء في الواجهة الأمامية — `js/lessonAssistant.js` يستدعيها تلقائياً عبر `supabase.functions.invoke(...)` باستخدام نفس `SUPABASE_URL`/`SUPABASE_ANON_KEY` الموجودين في `js/config.js`.

**6. (اختياري) تجربة الدالة محلياً قبل النشر**
```bash
supabase functions serve ask-lesson-ai --env-file ./supabase/.env.local
# ضع في الملف: MISTRAL_API_KEY=ضع_مفتاحك_هنا
```

### الحماية والتحكّم بالتكلفة
- **RLS + تحقق هوية:** الدالة ترفض أي طلب بدون تسجيل دخول صالح (JWT)، وتقرأ محتوى الدرس بصلاحية المستخدم نفسه (تحترم RLS)، فلا يمكن لأي شخص انتحال شخصية طالب آخر.
- **حدّ استخدام (Rate Limiting):** 8 أسئلة كحد أقصى لكل مستخدم كل 5 دقائق (قابل للتعديل من أعلى ملف `supabase/functions/ask-lesson-ai/index.ts`، الثابتان `RATE_LIMIT_MAX_REQUESTS` و`RATE_LIMIT_WINDOW_MINUTES`) لتفادي استنزاف رصيد Mistral API.
- **سجل الأسئلة:** كل سؤال وجواب يُسجَّل في جدول `lesson_ai_questions` (يستخدمه المشرف لاحقاً لمعرفة الأسئلة الأكثر تكراراً وتحسين محتوى الدروس)، مع استثناء تام لجدول `journal_entries` كالمعتاد.
- **تعقيم العرض:** الإجابة تُعرض عبر `escapeHtml()` في الواجهة لمنع أي XSS حتى لو أعاد النموذج نصاً غير متوقع.

### تخصيص النموذج
لتغيير نموذج Mistral المستخدم (افتراضياً `mistral-large-latest`) أو حدّ الأسئلة، عدّل الثوابت في أعلى ملف:
`supabase/functions/ask-lesson-ai/index.ts` ثم أعد النشر بـ `supabase functions deploy ask-lesson-ai`.

### 🩺 استكشاف الأخطاء وإصلاحها (لو ظهرت رسالة "خطأ غير متوقع")

**الخطوة الأولى دائماً: افتح سجلات الدالة لترى السبب الحقيقي**
```bash
supabase functions logs ask-lesson-ai
```
أو من Dashboard → **Edge Functions** → `ask-lesson-ai` → **Logs**. كل الأخطاء الآن مُسجَّلة هناك بالتفصيل (بعد التحديث الأخير)، كما أن الواجهة نفسها تعرض رسالة تشخيصية مختصرة بجانب أي خطأ.

**الأسباب الأكثر شيوعاً:**

| الأعراض | السبب المحتمل | الحل |
|---|---|---|
| خطأ فوري مع **كل** سؤال بلا استثناء | المشروع من نوع مفاتيح Supabase الحديث (`publishable_key`/`secret_key`) بدل القديم (`anon`/`service_role`) | تم إصلاحه في هذه النسخة: الدالة تجرّب الاسمين تلقائياً. تأكد أنك رفعت آخر نسخة من `index.ts` بأمر `supabase functions deploy ask-lesson-ai` |

## 🔔 نظام الإشعارات (Web Push)

إشعارات حقيقية تصل للمستخدم حتى لو كان التطبيق مغلقاً: (1) تلقائياً عند إضافة درس جديد، و(2) تذكيرات يدوية أو تلقائية يرسلها المشرف من تبويب **🔔 الإشعارات**.

### كيف يعمل تقنياً
```
المستخدم يفتح "حسابي" → يضغط "تفعيل الإشعارات" → يوافق المتصفح
                              ↓
        يُخزَّن اشتراكه (endpoint + مفاتيح تشفير) في push_subscriptions
                              ↓
عند إضافة درس جديد (تلقائياً) أو إرسال تذكير (يدوياً من المشرف)
                              ↓
     Supabase Edge Function (send-notification) تجيب الاشتراكات
     المطلوبة وترسل كل إشعار موقَّعاً بمفاتيح VAPID عبر بروتوكول Web Push
                              ↓
              service-worker.js يستقبل الحدث "push" ويعرض الإشعار
```
مفتاح VAPID الخاص (Private Key) **لا يوضع أبداً في كود الواجهة** — يبقى سرّاً على خادم Supabase فقط، تماماً كمفتاح Mistral API.

### خطوات التفعيل

**1. قاعدة البيانات** — نفّذ `schema_notifications.sql` في SQL Editor (بعد `schema.sql` و`schema_admin.sql`؛ ينشئ `push_subscriptions` و`notification_log`).

**2. ولّد مفاتيح VAPID** (مرة واحدة فقط لكل مشروع)
```bash
npx web-push generate-vapid-keys
```
سيعطيك هذا الأمر مفتاحين: `Public Key` و `Private Key`.

**3. ثبّت Supabase CLI واربطه بمشروعك** (إن لم تكن فعلت ذلك من قبل مع مساعد الطالب الذكي)
```bash
npm install -g supabase
supabase login
supabase link --project-ref YOUR-PROJECT-REF
```

**4. خزّن مفاتيح VAPID كأسرار (Secrets) على خادم Supabase**
```bash
supabase secrets set VAPID_PUBLIC_KEY=الصق_المفتاح_العام_هنا
supabase secrets set VAPID_PRIVATE_KEY=الصق_المفتاح_الخاص_هنا
supabase secrets set VAPID_SUBJECT=mailto:بريدك@example.com
```

**5. ضع المفتاح العام فقط في الواجهة** — افتح `js/config.js` وعدّل:
```js
VAPID_PUBLIC_KEY: "الصق_المفتاح_العام_نفسه_من_الخطوة_2",
```
(هذا المفتاح "عام" بطبيعته ولا خطورة في وضعه بكود الواجهة، بعكس المفتاح الخاص).

**6. انشر الدالة (Edge Function)**
```bash
supabase functions deploy send-notification
```

**7. جرّب الميزة**
- افتح التطبيق من حساب مستخدم عادي → **حسابي** → اضغط **تفعيل الإشعارات** ووافق على إذن المتصفح.
- من حساب مشرف: أضف درساً جديداً من تبويب **المحتوى** → يجب أن يصل إشعار "درس جديد! 📚" فوراً لكل من فعّل الإشعارات.
- أو من تبويب **🔔 الإشعارات**: أرسل تذكيراً يدوياً واختبر وصوله.

### إرسال تذكير يدوي أو لمن لم يُكمل درساً
من تبويب **🔔 الإشعارات** بلوحة الإدارة:
- **تذكير يدوي:** اكتب عنواناً ونصاً واختر "لكل المستخدمين" أو "لمستخدم معيّن" ثم أرسل.
- **تذكير غير النشطين:** حدّد عدد الأيام (افتراضياً 3) واضغط "إرسال الآن" — سيصل التذكير فوراً لكل من لم يُكمل أي درس خلال تلك المدة (بدون أي علاقة بجدول `journal_entries` الخاص).

### (اختياري) جدولة تذكير غير النشطين تلقائياً بدون تدخّل يدوي
لتشغيل تذكير "من لم يُكمل درساً منذ 3 أيام" تلقائياً كل يوم مثلاً، استخدم إضافتي `pg_cron` و`pg_net` المتوفرتين في Supabase (من **Database → Extensions** فعّل الاثنتين)، ثم في SQL Editor:
```sql
select cron.schedule(
  'daily-inactivity-reminder',
  '0 18 * * *', -- كل يوم الساعة 6 مساءً (بتوقيت UTC)
  $$
  select net.http_post(
    url := 'https://YOUR-PROJECT-REF.supabase.co/functions/v1/send-notification',
    headers := jsonb_build_object(
      'Content-Type', 'application/json',
      'Authorization', 'Bearer YOUR-SUPERVISOR-USER-JWT'
    ),
    body := jsonb_build_object(
      'type', 'auto_reminder',
      'title', 'لا تفوّت رحلتك! ⏰',
      'body', 'لديك دروس بانتظارك، أكمل رحلتك اليوم.',
      'target', 'inactive',
      'inactive_days', 3
    )
  );
  $$
);
```
**ملاحظة مهمة:** الدالة تتحقق أن المستدعي مشرف فعلاً، لذلك يلزم توكن (JWT) صالح لمستخدم `is_supervisor = true` في رأس `Authorization`. الأسهل عملياً تشغيل التذكير يدوياً من تبويب الإدارة عند الحاجة (كما في القسم أعلاه) إن كنت تفضّل تفادي تعقيد الجدولة عبر SQL.

### الحماية والتحكّم
- **صلاحية المشرف فقط:** الدالة (`send-notification`) ترفض أي طلب من مستخدم `is_supervisor != true`، حتى لو امتلك توكن دخول صالحاً.
- **تنظيف تلقائي:** أي اشتراك انتهت صلاحيته (المستخدم ألغى الإذن أو حذف المتصفح) يُحذف تلقائياً من `push_subscriptions` عند فشل الإرسال إليه.
- **خصوصية المذكرات:** لا تصل هذه الميزة إطلاقاً إلى جدول `journal_entries`، لا قراءةً ولا بناءً لأي إشعار من محتواه.
- **سجل شفاف:** كل إشعار (تلقائي أو يدوي) يُسجَّل في `notification_log` (النوع، النص، الوجهة، عدد المستلمين الفعلي، والتاريخ) ويظهر مباشرة في تبويب الإدارة.

## 🔒 ملاحظات أمنية مطبَّقة
- كل الحماية عبر **Row Level Security** في قاعدة البيانات، وليس فقط في الواجهة
- تعقيم كل نص يدخله المستخدم قبل عرضه (`js/utils/sanitize.js`) لمنع XSS
- المذكرات الخاصة محمية بسياسة RLS لا تسمح حتى للمشرف برؤيتها
- روابط واتساب تمر عبر `sanitizeUrl()` قبل تفعيلها
- **جديد**: تفعيل `Content-Security-Policy` في ملف `_headers` كطبقة حماية إضافية ضد XSS
  (تسمح فقط بـ: خطوط Google Fonts، اتصال Supabase (REST/Realtime)، تضمين يوتيوب عبر
  youtube-nocookie.com، والسكربتات من نفس الدومين فقط `'self'`). لو أضفت مستقبلاً أي
  مصدر خارجي جديد (CDN، خط آخر، خدمة صور)، يجب إضافته يدوياً لسطر CSP في `_headers`
  وإلا سيُحظر بصمت من المتصفح.

## 🔧 آخر إصلاح: اختفاء تبويبات (الرئيسية/مذكراتي/الدعم/حسابي/الإدارة) على الكمبيوتر

**العرض:** عند فتح التطبيق من متصفح كمبيوتر (شاشة عريضة)، كانت القائمة الجانبية بأكملها لا تظهر إطلاقاً — لا الرئيسية ولا مذكراتي ولا الدعم ولا حسابي ولا الإدارة — رغم أن كل شيء يعمل بشكل طبيعي تماماً على الجوال (الشريط السفلي).

**السبب الحقيقي (تم تشخيصه وإصلاحه في `css/layout.css`):** خطأ كلاسيكي في ترتيب قواعد CSS. كانت القاعدة `@media (min-width: 1024px) { .sidebar-nav { display: flex; } }` — التي يُفترض أن تُظهر الشريط الجانبي على الشاشات العريضة — مكتوبة **قبل** القاعدة الأساسية `.sidebar-nav { display: none; ... }` في نفس الملف. بما أن القاعدتين لهما نفس درجة التحديد بالضبط (محدِّد كلاس واحد `.sidebar-nav`)، فإن المتصفح يطبّق **الأحدث ظهوراً في ترتيب الملف** عند التعادل — وكانت هذه هي قاعدة `display: none`، فتُلغي قاعدة الإظهار دائماً بغض النظر عن عرض الشاشة الفعلي.

**الإصلاح:** أُعيد ترتيب القواعد بحيث تسبق القاعدة الأساسية (`display: none`) قاعدة الـ `@media` (`display: flex`) في ملف `css/layout.css`، فتعمل الآن كما صُمِّمت أصلاً: مخفية على الجوال (يظهر بدلها الشريط السفلي)، وظاهرة بجانب المحتوى على الكمبيوتر/الشاشات العريضة (1024px فأكثر) — بما فيها تبويب "🛠️ الإدارة" الذي يستخدم نفس تنسيق بقية التبويبات تماماً (`.nav-item`) فيظهر متناسقاً معها تلقائياً بمجرد إصلاح الظهور.

⚠️ **تذكّر رفع رقم `CACHE_NAME` في `service-worker.js`** بعد هذا التحديث (تم رفعه هنا فعلاً v6 → v7) حتى لا يستمر متصفح المستخدمين العائدين في تقديم نسخة `layout.css` القديمة المخزّنة مسبقاً.

## 🔧 تحديثات الإصلاح الأخيرة (اقرأ هذا القسم لو ظهرت "مشكلة غير متوقعة" عند الدخول)

### 1) لماذا كانت الرسالة تظهر أحياناً بدون سبب واضح؟
السبب الأكثر شيوعاً: **مشاريع Supabase المجانية تتوقف تلقائياً (Pause) بعد 7 أيام بدون أي نشاط حقيقي على قاعدة البيانات.** أي محاولة دخول أثناء التوقف تفشل فوراً.
- **تحقق الآن:** ادخل [supabase.com/dashboard](https://supabase.com/dashboard) وشوف هل المشروع مكتوب عليه "Paused" — لو كذلك اضغط **Restore project**.
- **حل دائم ومجاني:** فعّلنا لك ملف `.github/workflows/keep-alive.yml` يرسل طلباً بسيطاً لقاعدة بياناتك كل 3 أيام تلقائياً فيمنع التوقف نهائياً. لتفعيله:
  1. ارفع المشروع على GitHub (كما في خطوة "الرفع والنشر" أعلاه، هذا الملف موجود بالفعل ضمن الأكواد).
  2. من صفحة المستودع: **Settings → Secrets and variables → Actions → New repository secret**
  3. أضف `SUPABASE_URL` بقيمة رابط مشروعك، و`SUPABASE_ANON_KEY` بمفتاح anon public.
  4. هذا كل شيء — سيعمل تلقائياً بدون أي تدخل لاحق. يمكنك تجربته فوراً يدوياً من تبويب **Actions → Keep Supabase Project Awake → Run workflow**.
- **بديل:** الترقية لخطة Supabase Pro (25$/شهر) تشيل التوقف نهائياً — مفيدة فقط لو التطبيق عليه مستخدمون حقيقيون فعلاً.

### 2) إصلاحات في الكود (تمت بالفعل في هذه النسخة)
- **`js/utils/sanitize.js` (دالة `friendlyError`):** كانت أي حالة غير متوقعة (توقف المشروع، مشكلة شبكة، تجاوز حد المحاولات، جلسة منتهية، ملف شخصي مفقود...) تظهر كلها برسالة واحدة عامة "حدث خطأ غير متوقع" فتخفي السبب الحقيقي تماماً. الآن:
  - كل خطأ حقيقي يُطبع تفصيلياً في **Console المتصفح** (اضغط F12 → تبويب Console) حتى لو الرسالة المعروضة للمستخدم عامة — هذه أول خطوة تشخيص لأي مشكلة مستقبلية.
  - أُضيفت رسائل مخصصة لأكثر الحالات شيوعاً: انقطاع الاتصال/توقف المشروع، تكرار المحاولات، انتهاء الجلسة، عدم وجود الملف الشخصي، ونقص الصلاحيات.
- **`js/app.js` (نهاية الملف، `initAuthListener`):** كان فشل تحميل بيانات الحساب بعد تسجيل الدخول (لأي سبب) يترك المستخدم على **شاشة فارغة تماماً** (لا شاشة دخول ولا تطبيق) — وهذا بالضبط سبب "لا يدخل الحساب". الآن يُعرض الخطأ الحقيقي كرسالة، ويُعاد المستخدم تلقائياً وبأمان لشاشة تسجيل الدخول ليحاول من جديد.
- **`service-worker.js`:** تم رفع رقم النسخة (`CACHE_NAME`) مع تعليق توضيحي: **يجب رفع هذا الرقم يدوياً في كل مرة تُعدّل فيها `js/config.js` أو أي ملف JS/CSS**، وإلا سيستمر متصفح المستخدمين العائدين في استخدام نسخة قديمة مخزّنة (قد تحتوي مفاتيح Supabase قديمة) حتى بعد نشر التحديث.

### 3) قائمة تحقق سريعة لو استمرت المشكلة
- [ ] المشروع على Supabase **Active** وليس Paused

1. تفعيل **Realtime** على `ticket_messages` تلقائياً (بدل السطر اليدوي المنفصل الذي كان يُنسى بسهولة ويُسبب فشلاً صامتاً في الرد الفوري لمحادثة الدعم).
2. إنشاء دالتي تشخيص آمنتين (`admin_push_subscriptions_count`, `admin_realtime_status`) تُستخدمان في تبويب الإدارة الجديد "⚙️ الإعداد والتشخيص" — لا تكشفان أي بيانات حسّاسة، فقط عدّاد/حالة إجمالية.

بعد تنفيذه، ارفع أحدث نسخة من الملفات (Git push) — رقم كاش Service Worker مرفوع بالفعل هنا (`v7 → v8`).

### ✨ تحسينات تجربة المستخدم (الطالب)
- **تعديل الجنس لاحقاً من "حسابي":** كان يُحدَّد فقط عند التسجيل ولا يمكن تصحيحه بعدها إلا بتدخّل المشرف يدوياً من Supabase. الآن يوجد حقل + زر "حفظ الجنس" مباشرة في شاشة الحساب، ويُحدّث قائمة الدروس فوراً.
- **نافذة تهنئة عند إكمال 100% من مرحلة:** بدل الاعتماد الكامل على أن يكتشف المستخدم بنفسه شاشة "حسابي"، تظهر الآن نافذة تهنئة فور اكتمال كل دروس المرحلة الحالية، مع زر مباشر "الانتقال للمرحلة التالية". للمرحلة الأخيرة (الاستقرار الأسري) تظهر رسالة ختامية لإنهاء الرحلة كاملة بدل اقتراح انتقال (لا مرحلة بعدها). تُعرض مرة واحدة فقط لكل (مستخدم + مرحلة).
- **رسالة أوضح بعد إنشاء الحساب:** إن كان "تأكيد البريد الإلكتروني" مفعّلاً على مشروع Supabase، تظهر الآن رسالة صريحة "تحقّق من بريدك أولاً" بدل رسالة عامة كانت تجعل أول محاولة دخول تفشل دون تمهيد سابق يشرح السبب.

### ✨ تحسينات لوحة الإدارة
- **تبويب جديد "⚙️ الإعداد والتشخيص":** يفحص تلقائياً (VAPID مضبوط؟ عدد المشتركين بالإشعارات، هل Realtime مفعّل فعلياً على الدعم الفني، رقم واتساب مخصّص؟) ويعرض حالة كل بند بوضوح (✅/⚠️/❔)، بدل اكتشاف الأعطال بالتجربة الفعلية أو قراءة README بعناية. مفتاح Mistral ومفاتيح VAPID الخاصة تبقى تحققاً يدوياً عمداً لأنها أسرار خادم لا يجوز فحصها من المتصفح أمنياً.
- **تنبيه واضح عند فشل إشعار "درس جديد":** سابقاً كان الفشل (مثلاً Edge Function غير منشورة) يُطبع فقط في Console الذي لا يراه المشرف، فيظن أن الإشعار وصل بينما لم يصل لأحد. الآن يظهر تنبيه Toast ثانوي واضح للمشرف فور فشل الإرسال، دون أن يمنع ذلك نجاح حفظ الدرس نفسه.
